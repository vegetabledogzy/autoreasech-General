# autoiter — 通用科研模型自主迭代指令

这是一个通用的 AI agent 自主科研迭代框架。你的任务是：不断改进模型，让目标指标持续提升，并记录每次改动的思路与反思。

## 核心理念

与 autoresearch 不同，本框架：
- **不绑定固定训练时间**：每次实验运行到自然结束（收敛、达到 epoch 上限、或触发 early stopping）
- **指标驱动**：你的目标是让 `config.yaml` 中定义的指标持续改善
- **反思优先**：每次实验后必须写反思，分析为什么有效/无效，指导下一步方向

## Setup

与用户协商完成以下步骤：

1. **确认任务**：用户需要在 `config.yaml` 中填写：
   - 任务描述（分类/检测/生成/回归/...）
   - 目标指标名称（如 accuracy、mAP、BLEU、loss 等）
   - 指标方向（higher_is_better 或 lower_is_better）
   - 可编辑文件列表（默认只有 `train.py`）
   - 运行命令（如 `python train.py`）
   - 评估命令（如 `python evaluate.py`，或与训练合并）
   - 最大单次实验时长（超时视为失败）

2. **约定 run tag**：基于日期提议 tag（如 `mar17`），创建分支 `autoiter/<tag>`。

3. **阅读项目文件**：
   - `config.yaml` — 实验配置
   - `evaluate.py` — 评估脚本（只读，除非用户允许修改）
   - `train.py` — 你要迭代的训练脚本
   - 用户指定的其他可读文件

4. **初始化日志**：创建 `results.tsv` 和 `reflections.md`。

5. **确认并开始**。

## 可编辑范围

- 默认只修改 `config.yaml` 中 `editable_files` 列出的文件
- **不得修改** `evaluate.py`（除非用户明确允许）
- **不得新增依赖**（除非用户明确允许）
- 一切改动必须 git commit，保持可追溯

## 实验循环

LOOP FOREVER:

### 1. 思考（Think）
在动手之前，先回顾：
- 当前最佳指标是多少？
- 上一次实验的反思结论是什么？
- 基于已有实验历史，下一步最有价值的方向是什么？

将你的实验假设写入 `reflections.md` 的新条目（在运行之前）：
```
## Experiment #N: <简短标题>
**假设**: <你认为这个改动会有效的原因>
**改动**: <具体要改什么>
```

### 2. 改动（Modify）
- 修改可编辑文件，实现你的实验想法
- git commit，message 简洁描述改动

### 3. 运行（Run）
```bash
<config.yaml 中的 run_command> > run.log 2>&1
```
- 将所有输出重定向到 run.log，不要让输出淹没你的上下文
- 如果超过 `max_experiment_time`，kill 并视为失败

### 4. 评估（Evaluate）
- 运行评估命令（如果与训练分离的话）
- 从 run.log 或评估输出中提取目标指标：
```bash
grep "<config.yaml 中的 metric_pattern>" run.log
```

### 5. 判定（Judge）
- 如果指标改善：**keep**，保留 commit，更新最佳记录
- 如果指标持平或退步：**discard**，`git reset --hard` 回退到上一个 keep 的 commit
- 如果崩溃：**crash**，尝试修复简单 bug；如果根本性问题则跳过

### 6. 记录（Log）
在 `results.tsv` 中追加一行：
```
commit	metric_value	status	description
```

### 7. 反思（Reflect）
**这是与 autoresearch 最大的区别。** 在 `reflections.md` 中补充本次实验的结论：
```
**结果**: keep/discard/crash, metric = xxx
**分析**: <为什么有效/无效？与假设是否一致？>
**启发**: <这个结果对后续实验有什么指导意义？>
**下一步**: <基于此反思，下一个最值得尝试的方向>
```

### 8. 回到步骤 1

## 反思系统详细说明

`reflections.md` 是你的科研笔记本。它的价值在于：

1. **避免重复**：记录失败的方向，避免走回头路
2. **积累洞察**：多次实验后，你会发现模式（如"这个模型对 LR 很敏感"）
3. **指导方向**：每次反思的"下一步"直接驱动下一轮实验
4. **人类可读**：用户醒来后可以快速了解你做了什么、学到了什么

反思的质量比数量重要。不要写废话，写真正有信息量的分析。

## 停滞策略

如果连续 3 次实验都没有改善，触发"深度反思"：
1. 重新阅读所有反思记录，寻找被忽略的模式
2. 重新阅读可编辑文件，寻找新的优化角度
3. 考虑更激进的改动（架构变更、不同的训练策略等）
4. 在 `reflections.md` 中写一段"阶段性总结"，梳理已知信息

如果连续 5 次都没有改善，在 `reflections.md` 中写一段详细的"瓶颈分析"，但**不要停下来**。继续尝试。

## 简洁性原则

与 autoresearch 一致：
- 同等效果下，更简洁的代码更好
- 微小改善 + 大量复杂代码 = 不值得
- 删代码且效果不变 = 好结果
- 大幅改善 + 适度复杂 = 值得

## 超时与崩溃处理

- **超时**：超过 `max_experiment_time` 则 kill，记录为 crash
- **崩溃**：简单 bug（typo、import 错误）→ 修复重跑；根本性问题 → 跳过，记录 crash
- 连续 crash 超过 3 次 → 回退到最后一个 keep 状态，换方向

## NEVER STOP

实验循环开始后，**不要停下来问用户**。用户可能不在电脑前。你是自主的研究者。如果没有想法了，重读反思记录，重读代码，想得更深。循环直到被手动中断。
